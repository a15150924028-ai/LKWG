# 棋契陛下八形态数据设计

## 目标

将本地数据中单个合并的 `棋契陛下` 拆分为八个互相独立的最终形态。每个形态必须拥有独立精灵 ID、独立技能池数组和独立进化链，后续可以单独修正任意技能池而不影响其他形态。

## 最终形态

1. `棋契陛下（棋骑士白子）`
2. `棋契陛下（棋骑士黑子）`
3. `棋契陛下（棋齐垒白子）`
4. `棋契陛下（棋齐垒黑子）`
5. `棋契陛下（棋祈督白子）`
6. `棋契陛下（棋祈督黑子）`
7. `棋契陛下（棋绮后白子）`
8. `棋契陛下（棋绮后黑子）`

名称中的空格只用于用户描述辨识，写入数据时不保留括号内多余空格。

## 技能池

- 八个最终形态都必须写入独立的 `skillIds` 数组。
- 第1条白子和黑子使用各自页面显示的技能池。
- 第2、3、4条当前只有每条一个页面数据，因此同一条的白子和黑子暂时复制相同技能内容。
- 即使暂时内容相同，也不得让两个形态引用同一可变数组或再次合并为一个精灵记录。
- 后续用户补充白黑差异时，只修改对应形态的 `skillIds`。
- 不在本地数据、注释、状态文案或 UI 中保存资料页面链接。

## 进化链

八条进化链分别建立，不共享最终节点：

- 棋棋（白子） → 棋骑士（白子） → 棋契陛下（棋骑士白子）
- 棋棋（黑子） → 棋骑士（黑子） → 棋契陛下（棋骑士黑子）
- 棋棋（白子） → 棋齐垒（白子） → 棋契陛下（棋齐垒白子）
- 棋棋（黑子） → 棋齐垒（黑子） → 棋契陛下（棋齐垒黑子）
- 棋棋（白子） → 棋祈督（白子） → 棋契陛下（棋祈督白子）
- 棋棋（黑子） → 棋祈督（黑子） → 棋契陛下（棋祈督黑子）
- 棋棋（白子） → 棋绮后（白子） → 棋契陛下（棋绮后白子）
- 棋棋（黑子） → 棋绮后（黑子） → 棋契陛下（棋绮后黑子）

每条链使用独立 `chainId`，并同步更新初始形态、中间形态和最终形态的进化元数据。

## 特性与 PVP

- 八个形态的数据特性均保留 `御驾亲征`。
- `棋契陛下（棋绮后白子）` 和 `棋契陛下（棋绮后黑子）` 在 PVP 特性层数规则中额外享受 `渗透`：
  - 每层物攻、魔攻、物防、魔防各增加 5%。
- 其余六个形态不自动套用 `渗透` 层数效果。
- 两个棋绮后最终形态仍显示最终形态名称和 `御驾亲征` 数据，不改名为棋绮后。

## 数据迁移

- 删除旧的单个 `monster-棋契陛下` 合并记录。
- 清理旧记录被其他进化元数据引用的位置。
- 创建八个稳定且唯一的新 ID。
- 复用现有本地图标、属性和 `御驾亲征` 特性记录，不引入外部资源链接。

## 验证

新增回归测试验证：

1. 本地恰好存在八个命名明确的棋契陛下最终形态。
2. 旧合并记录不存在。
3. 八个 ID、八个 `chainId` 和八个 `skillIds` 数组互相独立。
4. 八组种族值符合上表。
5. 八条进化链的颜色和中间形态正确。
6. 第2、3、4条白黑技能内容当前相同，但记录保持独立。
7. 仅两个棋绮后最终形态获得每层双攻双防 5% 的渗透效果。
8. 所有本地静态测试和内联脚本解析继续通过。
